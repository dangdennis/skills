Extensively research the current state of this project and produce a comprehensive TODO list. Follow these steps:

## Step 1: Understand the Project

Investigate what this project is and what it's trying to achieve:

1. Read any planning docs, README, PLAN.md, CLAUDE.md, ARCHITECTURE.md, CONTRIBUTING.md, or similar files in the repo root.
2. Check the project's language/framework by examining build files (Cargo.toml, package.json, pyproject.toml, go.mod, Makefile, etc.).
3. Review git log to understand the history and trajectory of development.
4. Look for any existing TODO files, issue trackers, or task lists.

## Step 2: Audit Current State

Thoroughly investigate what has been implemented so far:

1. **Project structure**: Map out the directory structure and understand the organization.
2. **Source code**: For every module/package/crate, audit its contents — are there real implementations or just stubs? What functions/types/classes are defined vs missing?
3. **Tests**: What tests exist? Are they passing? Run the project's test command if a build system is set up.
4. **CI/CD**: Check for GitHub Actions, CI configs, or any automation.
5. **Dependencies**: Check dependency files for what's already pulled in vs what might still be needed.
6. **Build status**: Can the project build/compile successfully right now?

## Step 3: Cross-Reference Intent vs Reality

Based on whatever planning docs, README, or project structure exists, determine:
- What items are **done** (implemented and working)
- What items are **partially done** (started but incomplete)
- What items are **not started**
- What items are **blocked** (waiting on prerequisites)

If there are defined phases or milestones, evaluate progress against each one.

## Step 4: Produce the TODO List

Create or update a `TODO.md` file in the project root with the following structure:

```markdown
# TODO

> Auto-generated by /todos on [date]. Re-run to refresh.

## Project Status Summary
[2-3 sentence overview of where the project stands]

## Current Milestone
[What milestone/phase the project is currently in based on what's been completed]

## Completed Work
- [x] Item (with brief note on what was done)

## In Progress
- [ ] Item — [status note, what remains]

## Up Next
- [ ] Concrete actionable item 1
- [ ] Concrete actionable item 2
...

## Blocked
- [ ] Item — blocked on: [dependency]

## Future Work
### [Milestone/Feature Area Name]
- [ ] Key deliverable 1
- [ ] Key deliverable 2
...

## Technical Debt & Notes
- Any issues, risks, or observations discovered during the audit
```

### Guidelines for writing TODOs:
- Each TODO should be a **concrete, actionable task** that an agent or developer can pick up and execute independently.
- Break large items into smaller subtasks where possible.
- Include file paths where work needs to happen when known.
- Note any decisions that need to be made (e.g., library choices, API design questions).
- Order items by priority within each section — what should be done first to unblock the most work.
- Include milestone/gate criteria so it's clear when a phase is "done".
- Be specific: "Implement X in src/foo.rs" is better than "Work on foo module".

## Step 5: Summary

After creating TODO.md, provide a brief summary of:
- The current project milestone/phase
- How many TODOs were identified
- The top 3 highest-priority items to work on next
